## 背景

根据 requirements/0001 的检查报告，当前实现存在三个核心偏差：扫描结果未区分新旧安装、`scan_interval_hours` 未生效、`show_notifications` 未生效。本设计文档定义第一期修改方案，解决这些偏差。

## 目标与范围

- 让"每天安装的第三方包"在产品语义上成立
- 实现配置项 `scan_interval_hours` 和 `show_notifications`
- 保持旧数据兼容和现有功能不回退

### 本期包含

- 包清单快照存储与差异计算
- 每日安装记录保存"相对上次快照新增的包"
- `root.after()` 定时扫描调度
- `show_notifications` 控制弹窗
- 基础验证测试

### 本期不包含

- 系统托盘、后台常驻、开机自启动
- macOS 原生通知中心集成
- pip 多 Python 环境扫描
- Homebrew cask、Go 版本识别
- UI 视觉重构

## 设计方案

### 1. 数据模型

新结构兼容旧格式：

```json
{
  "records": {
    "2026-05-19": [
      {
        "manager": "pip",
        "name": "jinja2",
        "version": "3.1.0",
        "time": "2026-05-19 10:30"
      }
    ]
  },
  "snapshot": {
    "packages": [
      {"manager": "pip", "name": "jinja2", "version": "3.1.0"}
    ],
    "scanned_at": "2026-05-19 10:30"
  },
  "metadata": {
    "schema_version": 2,
    "last_scan_at": "2026-05-19 10:30"
  }
}
```

**兼容策略**：读到旧格式时按旧 `records` 加载，首次保存时迁移，旧记录保留不删除。

### 2. 新增安装识别

差异计算流程：

1. 扫描所有启用包管理器，得到当前全量包列表
2. 从存储中读取上一份快照
3. 用 `(manager, name)` 作为包身份键
4. 当前快照中存在、上一快照中不存在的包视为新增安装
5. 新增包写入当天 `records[date]`
6. 更新快照和 `last_scan_at`

**首次扫描**：只建立基线，不把全量包写入当天记录。

### 3. 定时扫描

在 `PackageMonitorUI` 中实现：

- GUI 启动后根据 `scan_interval_hours` 注册定时任务
- 周期任务到期后调用现有扫描线程
- 扫描完成后再次注册下一次扫描
- 最小间隔保护（不低于 1 小时）

**并发保护**：使用 `is_scanning` 字段互斥，避免手动扫描和定时扫描同时运行。

### 4. 通知开关

`show_notifications` 控制当前 messagebox：

- 扫描完成弹窗受该配置控制
- 报告生成弹窗受该配置控制
- 状态栏仍正常更新

## 任务拆分

| 任务 | 涉及文件 | 验收标准 |
|---|---|---|
| P1-1: 存储结构升级 | `storage.py` | 旧格式可读，新格式保存，无记录时正常启动 |
| P1-2: 新增安装差异计算 | `storage.py`, `main.py`, `ui.py` | 首次扫描不写入全量，第二次无新增时不增加 |
| P1-3: 实现 `scan_interval_hours` | `ui.py` | 配置为 24 时安排下一次扫描，手动扫描不并发 |
| P1-4: 实现 `show_notifications` | `ui.py` | false 时不弹窗，状态栏仍显示 |
| P1-5: 基础验证 | `tests/` | 静态编译通过，差异计算验证通过 |

## 风险与处理

| 风险 | 影响 | 处理方式 |
|---|---|---|
| 首次扫描不显示全量包 | 中 | 状态栏提示"已建立基线" |
| 旧格式迁移出错 | 高 | 加载时兼容，保存前不删旧记录 |
| 定时/手动扫描并发 | 中 | `is_scanning` 互斥 |
| 包管理器命令超时 | 中 | 保持 30 秒 timeout |
