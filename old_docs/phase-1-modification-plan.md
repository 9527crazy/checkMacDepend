# 第一期修改规划

规划日期：2026-05-19

## 目标

第一期目标是让当前实现更准确地满足 README 的核心描述：监控和展示每天安装的第三方包。

本期优先解决三个已确认偏差：

1. 扫描结果应识别“新增安装包”，而不是把当前全部已安装包都记为当天安装。
2. `scan_interval_hours` 配置应生效。
3. `show_notifications` 配置应生效。

## 范围

### 本期包含

- 增加包清单快照存储，用于比较本次扫描和上次扫描的差异。
- 将每日安装记录改为保存“相对上次快照新增的包”。
- 保留现有 JSON 存储方式和 GUI 主流程。
- 在 GUI 中使用 `root.after()` 根据 `scan_interval_hours` 触发周期扫描。
- 用 `show_notifications` 控制扫描完成、报告生成等弹窗提示。
- 补充面向存储和差异计算的基础测试或可重复验证脚本。

### 本期不包含

- 不做完整系统托盘、后台常驻或开机自启动能力。
- 不做 macOS 原生通知中心集成。
- 不扩展所有 pip 虚拟环境扫描。
- 不做 Homebrew cask、Go 版本识别等扫描器增强。
- 不重构 UI 视觉设计。

## 设计方案

### 1. 数据模型调整

当前 `records.json` 只保存按日期分组的包记录：

```json
{
  "2026-05-19": [
    {
      "manager": "pip",
      "name": "jinja2",
      "version": "3.1.0",
      "time": "2026-05-19 10:30"
    }
  ]
}
```

建议第一期引入兼容性较好的新结构：

```json
{
  "records": {
    "2026-05-19": []
  },
  "snapshot": {
    "packages": [
      {
        "manager": "pip",
        "name": "jinja2",
        "version": "3.1.0"
      }
    ],
    "scanned_at": "2026-05-19 10:30"
  },
  "metadata": {
    "schema_version": 2,
    "last_scan_at": "2026-05-19 10:30"
  }
}
```

兼容策略：

- 如果读到旧格式，即顶层是日期字典，则按旧 `records` 加载。
- 首次保存时迁移为新格式。
- 旧记录保留，不清空用户已有数据。
- 新增快照只用于后续差异判断；首次扫描建立基线，默认不把全量包写入当天安装记录。

### 2. 新增安装识别

新增一个差异计算流程：

1. 扫描所有启用包管理器，得到当前全量包列表。
2. 从存储中读取上一份快照。
3. 用 `(manager, name)` 作为包身份键。
4. 当前快照中存在、上一快照中不存在的包视为新增安装。
5. 新增包写入当天 `records[date]`，并将 `time` 设置为扫描完成时间。
6. 无论是否发现新增包，都更新快照和 `last_scan_at`。

版本变化处理：

- 第一期暂不把版本变化视为新增安装。
- 可在记录中保留当前版本，便于未来扩展“升级记录”。

首次扫描处理：

- 首次扫描只建立基线，不把所有已安装包标记为当天安装。
- UI 提示文案建议为“首次扫描已建立基线，后续扫描将记录新增安装包”。

### 3. 定时扫描

实现位置建议放在 `PackageMonitorUI`：

- GUI 启动后根据 `config.get("scan_interval_hours")` 注册定时任务。
- 周期任务到期后调用现有扫描线程。
- 扫描完成后再次注册下一次扫描。
- 最小间隔建议做保护，例如低于 1 小时按 1 小时处理，避免误配置造成频繁执行。

需要避免的问题：

- 用户手动扫描和定时扫描同时运行。
- 扫描线程直接更新 tkinter 控件。

建议增加状态字段：

- `self.is_scanning`
- `self.next_scan_job`

### 4. 通知开关

`show_notifications` 第一期先控制当前 messagebox：

- 扫描完成弹窗受该配置控制。
- 报告生成弹窗受该配置控制。
- 状态栏仍然正常更新，不受该配置影响。

这样能兑现 README 中的配置项，同时不引入新的平台依赖。

## 任务拆分

### P1-1：存储结构升级

涉及文件：

- `storage.py`

工作内容：

- 支持旧格式和新格式加载。
- 增加 `get_snapshot()`、`update_snapshot()`、`get_last_scan_at()` 等方法。
- 保持现有 `get_packages()`、`get_all_dates()`、`search_packages()` 对外行为不变。
- 保存时写入 `schema_version`。

验收标准：

- 旧格式 `records.json` 可正常读取。
- 新格式保存后仍能按日期浏览历史记录。
- 无记录时程序可正常启动。

### P1-2：新增安装差异计算

涉及文件：

- `storage.py`
- `main.py`
- `ui.py`

工作内容：

- 增加“扫描全量包 -> 对比快照 -> 保存新增包 -> 更新快照”的统一方法。
- 启动扫描和手动扫描共用同一逻辑。
- 扫描完成展示新增包数量，而不是全量包数量。

验收标准：

- 首次扫描不把全量已安装包写入当天记录。
- 第二次扫描无新增包时，当天记录不增加。
- 模拟多出一个包时，当天记录只增加该包。

### P1-3：实现 `scan_interval_hours`

涉及文件：

- `ui.py`
- 可能涉及 `config.py`

工作内容：

- GUI 初始化后注册下一次扫描。
- 扫描完成后继续安排下一轮扫描。
- 手动扫描期间跳过定时扫描或复用同一队列保护。

验收标准：

- 配置为 24 时，启动后会安排下一次 24 小时后扫描。
- 配置缺失时使用默认值。
- 手动扫描和定时扫描不会并发执行。

### P1-4：实现 `show_notifications`

涉及文件：

- `ui.py`

工作内容：

- 封装 `_should_show_notifications()`。
- 扫描完成、报告生成时按配置决定是否弹窗。

验收标准：

- `show_notifications: false` 时，不出现完成弹窗。
- 状态栏仍显示扫描和报告结果。

### P1-5：基础验证

涉及文件：

- 可新增 `tests/` 或 `docs/phase-1-verification.md`

工作内容：

- 针对差异计算写最小测试，优先覆盖首次扫描、无新增、新增包三个场景。
- 执行 `python3 -m py_compile main.py config.py storage.py scanner.py report.py ui.py`。
- 如引入测试文件，执行对应测试命令。

验收标准：

- 静态编译通过。
- 差异计算验证通过。
- README 核心需求检查中的主要未满足项下降。

## 推荐实施顺序

1. 先改 `storage.py`，完成兼容加载和快照保存。
2. 抽出统一扫描保存逻辑，让启动扫描和手动扫描使用同一条路径。
3. 修改 UI 文案，从“发现 N 个已安装的包”改为“发现 N 个新增安装包”。
4. 增加定时扫描调度。
5. 接入 `show_notifications`。
6. 补充验证文档或测试。

## 风险与处理

| 风险 | 影响 | 处理方式 |
|---|---|---|
| 首次扫描不再显示全量包，用户可能以为没扫描到 | 中 | 状态栏明确提示“已建立基线”。 |
| 旧 `records.json` 格式迁移出错 | 高 | 加载时保留旧结构兼容，保存前避免删除旧记录。 |
| 定时扫描和手动扫描并发 | 中 | 用 `is_scanning` 做互斥。 |
| 包管理器命令慢或卡住 | 中 | 继续使用现有 30 秒 timeout，本期不扩大范围。 |
| 版本升级不被记录 | 低 | 本期明确只记录新增安装，升级记录留到后续。 |

## 完成定义

第一期完成后，应满足：

- README 中“每天安装的第三方包”在产品语义上可成立。
- 用户每天看到的是扫描间隔内新增安装的包，而非系统所有已安装包。
- `scan_interval_hours` 和 `show_notifications` 不再只是空配置。
- 旧数据仍能读取，历史日期浏览、搜索、报告生成不回退。
- 静态编译和差异计算验证通过。
